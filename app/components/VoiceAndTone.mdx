# Voice & Tone

## The metaphor

The Ledger is a newspaper. Not metaphorically, as a styling choice — literally, as an editorial position. A newspaper has a voice: institutional, authoritative, and scrupulously disinterested. It reports what happened. It does not celebrate, commiserate, or editorialize. It describes the state of things.

That voice governs every word in The Ledger — from section headlines to error messages. The metaphor is not decoration; it is the standard against which all copy is measured.

## The governing principle

A newspaper does not tell you how to feel about the news. It tells you what is true.

**The Ledger describes what is. It does not editorialize about it.**

Net worth went up. Net worth went down. These are facts. The application reports them without inflection — it does not congratulate when the number rises or soften the blow when it falls. It prints the number and moves on.

This has practical consequences for every category of copy in the application.

## Institutional voice

Newspapers do not write to a single reader. They publish. The voice is institutional — it states facts without a subject, without warmth, without the assumption that the reader needs to be managed.

Avoid possessive framing in data headlines and section labels. The data belongs to the reader by context; the possessive is redundant and softens the declarative register.

Second person is not forbidden. It is appropriate in instructions, form labels, and direct address ("Enter your password," "Confirm with your authenticator code"). The rule applies to data presentation and section labels, where the institutional register is expected.

## Data callouts

Data callouts use a compact, headline-fragment style: direction, amount, and timeframe — no subject, no verb. This is the register of a financial table or a stock ticker: precise, scannable, and complete without being conversational.

The pattern: **[direction] [amount] [timeframe].**

Write figures in full. Abbreviations like `$31k` or `~$4K` belong in informal contexts; the newspaper prints the number. The `Currency` component handles display formatting — always pass a value in cents and let the component render it.

Timeframes are written out in plain language: "over the last 30 days," "since the beginning of this year," "over the last year." Do not abbreviate to "YTD" or "30d."

## Error messages

Errors name the problem precisely and without apology. A newspaper does not apologize for printing a correction — it states what is wrong and moves on.

The pattern: **[what is wrong].** Not: _[expression of regret] + [vague description] + [encouragement to try again]._

If the error is recoverable, state what can be done. If it is not, state that plainly. Do not soften the message with reassurance.

## The unauthenticated surface

The homepage and marketing copy follow the same discipline. The publication does not have a personal voice — it does not speak as an individual, and it does not use first person.

First-person narrative ("I built the finance tool I couldn't find") is inconsistent with the institutional register. It positions the copy as a personal statement rather than a factual account of a product that exists.

The unauthenticated surface should read as a prospectus: factual, direct, written in the third person as though describing something that exists rather than selling it.

### Features vs. benefits

The distinction to hold onto: describe what the user *gets*, not how it works. Implementation details ("daily syncing via Plaid," "balance carry-forward logic," "SQLite") answer engineering questions, not user questions. The copy should answer: what does this give me?

"Self-hosted" is a specific failure mode here. It implies the user is running the server themselves — which is not the case. Drop it entirely. The meaningful thing underneath it is that the application is built to last without a subscription or a company's continued interest in operating it. Say that.

Technical details can survive translation when they have a clear user-facing consequence. "No subscription" is a benefit, not a feature. "No ads. Data isn't sold." are both factual and user-relevant. Keep those. Cut the ones that don't answer "so what?"

## What to avoid

These patterns are common in consumer software and wrong for The Ledger.

**Enthusiasm.** No exclamation points in normal operation. Financial data moving in any direction is not an occasion for celebration in the copy. An exclamation point on a success message reads as the application congratulating itself.

**Apology and hedging.** "Sorry for the inconvenience." "It looks like something may have gone wrong." "We're having trouble." The newspaper does not hedge. It states what is known.

**Informal register.** "Submit that bad boy," "Hey there," "Uh oh" — none of these. Emoji used as copy falls into the same category. The Ledger is a publication, not a chat interface.

**Promotional copy in the UI.** "Unlock your financial potential." "See what's possible." These belong in advertising. The UI describes what exists.

**Motivation and encouragement.** "Keep it up!" "You're on track!" "Great job setting up 2FA!" The application reports facts. It does not coach.

**Vagueness where specificity is possible.** "Something went wrong." Name the problem when it is known. Vague error messages are a failure of reporting.
